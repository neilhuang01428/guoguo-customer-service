# 評分與收斂

## 硬性品質閘門（四道，任一不過就先合併/研究/淘汰）

- **GATE 1 事實與安全**：能否說清楚型號、OS、地區、證據與停止自行處理點？
- **GATE 2 去重與粒度**：是否與既有題共享同一根本答案？是否值得單獨一篇？
- **GATE 3 讀者增量**：是否提供判斷、流程、測試或在地情境，而不只是改寫手冊？
- **GATE 4 題庫平衡**：是否補到目前欠缺的產品、歷程、人物或搜尋意圖？

## 12 項加權評分（0–5 分）

| 項目 | 權重 | 問句 |
|---|---|---|
| 痛點強度 | +2 | 不解決會損失資料、時間、金錢或信心嗎？ |
| 出現頻率 | +1.5 | 自有資料與搜尋訊號是否反覆出現？ |
| 受眾貼合 | +1.5 | 是否明確服務你的 Apple 新品／福利機顧客？ |
| 證據強度 | +2 | 有官方來源、實機或第一方資料可支持嗎？ |
| 原創增量 | +2 | 能提供別人沒有的測試、門檻、圖解或案例嗎？ |
| 決策價值 | +1.5 | 看完能做出買、修、設、換或停止的決定嗎？ |
| 長尾壽命 | +1 | 問題是否跨版本仍存在？ |
| 時效急迫 | +1 | 現在是否有版本、服務或市場變動？ |
| 群集槓桿 | +1 | 能否補內容缺口並連到多篇相關文章？ |
| 答案信心 | +2 | 適用型號、版本、地區與例外是否說得清楚？ |
| 製作成本 | −1 | 需要大量實測、專家或長時間設備嗎？（越高越扣分） |
| 更新負擔 | −1 | UI、價格、支援範圍是否很快過期？（越高越扣分） |

> 注意：「製作成本」與「更新負擔」是負權重項目，分數越高扣分越多。

## 處理建議（七種）

- `WRITE_NOW`：立即撰寫
- `EVERGREEN_QUEUE`：常青佇列
- `TIMELY_QUEUE`：時效佇列
- `EXPERIMENT`：需先實測
- `RESEARCH_FIRST`：先研究再定案
- `MERGE`：合併到既有題
- `DROP`：淘汰

## 題庫配額假設

| 類別 | 佔比 | 說明 |
|---|---|---|
| 疑難排除與日常痛點 | 35% | 高需求、高信任；優先用客服與搜尋訊號。 |
| 使用技巧與工作流 | 20% | 強調完成任務，不做泛泛『必學技巧』。 |
| 選購、新舊機與福利機 | 20% | 用適配門檻與證據取代規格堆砌。 |
| 細節認知、迷思與安全 | 15% | 修正錯誤心智模型，建立長期信任。 |
| 版本、服務與市場變動 | 10% | 事件驅動，必須綁查核日與更新觸發。 |

## 八個反模式

1. **盲目排列組合**：34 個維度不是要全部相乘；沒有真實摩擦、取捨或後果的交叉一律淘汰。
2. **先想 SEO 標題**：會把同一答案包成幾十個問句。先保存問題原子，最後才命名。
3. **只改機型名稱**：若 UI、限制、根因與行動完全相同，應做一篇範圍清楚的文章。
4. **重寫官方手冊**：至少補情境、選擇門檻、失敗路徑、台灣限制或第一手證據。
5. **把社群留言當事實**：留言用來發現症狀與語言；功能結論必須回官方或實機。
6. **忽略版本與地區**：同名功能可能因 OS、機型、語言與地區不同；每題都要有適用範圍。
7. **只有正向購買內容**：可信的內容也要回答誰不適合、何時不值得加價、何時應直接送修。
8. **總分決定一切**：分數是討論工具；高需求但無法安全查證的題目仍應暫緩。
