# 訊號來源與證據

## 六層訊號地圖

### 第一方營運 — 最接近你的真實顧客

- **包含**：銷售對話、客服工單、站內搜尋、退貨原因、保固問題、商品頁 QA、檢測紀錄
- **轉換方式**：原話 → 發生時點 → 已嘗試方法 → 未解決決定 → 問題原子
- **注意**：商業價值高；樣本偏向既有客群，需要去識別化。

### 官方事實 — 用來確認功能與邊界

- **包含**：Apple 支援、使用手冊、技術規格、比較頁、版本安全更新、維修方案、過時／停產清單、系統狀態
- **轉換方式**：新舊差分 → 受影響型號／地區 → 使用者行動改變 → 新文或更新
- **注意**：可信度高，但官方通常不替你回答台灣福利機買家的情境。

### 搜尋需求 — 看人們用什麼話找答案

- **包含**：Search Console、Google Trends、自動完成、People Also Ask、相關搜尋、YouTube 搜尋建議
- **轉換方式**：查詢詞 → 搜尋意圖 → 根本答案 → 是否已有完整解答
- **注意**：有需求不等於值得寫；先過可驗證與差異化門檻。

### 社群聲音 — 發現官方文件看不到的摩擦

- **包含**：Apple 支援社群、Mobile01、PTT、Dcard、Reddit、YouTube 留言、社團討論
- **轉換方式**：大量貼文 → 症狀／版本／機型聚類 → 重複度 → 回官方與實機查證
- **注意**：適合找線索；單一留言不能當產品事實。

### 市場接縫 — 找配件、電信商與 App 的落差

- **包含**：電信商說明、App 更新紀錄、配件規格與負評、教育／企業部署文件、競品 FAQ
- **轉換方式**：宣稱或抱怨 → 相容條件 → 例外 → 使用情境 → 驗證題
- **注意**：留意國家、版本與商業偏誤。

### 第一手實測 — 建立別人難複製的證據

- **包含**：跨型號操作、固定條件續航、相容性檢查、螢幕錄影、新手觀察、售後案例
- **轉換方式**：爭議問題 → 可重複步驟 → 控制變因 → 結果與限制 → 教學或決策門檻
- **注意**：原創價值最高；先定義方法，避免把單機結果過度外推。

## 證據等級

| 等級 | 定義 |
|---|---|
| E3 | 多個直接顧客/內部訊號 |
| E2 | 重複外部訊號 |
| E1 | 依使用流程合理推導 |
| E0 | 純探索假設 |

**不可把 E0 或單一個案描述成普遍現象。**

## 一個完整發想示範

1. **01 保存原話**：『我只是放一晚就掉很多電，這台福利 iPad 是不是電池有問題？』先不改寫成電池故障。
2. **02 拆問題原子**：人物＝剛收到福利機；階段＝驗收；症狀＝待機掉電；情緒＝怕買到問題機；決定＝是否正常／是否退換。
3. **03 建症狀樹**：區分待機或使用中、單次或連續、更新／還原後、Wi‑Fi 或行動網路、是否伴隨發熱或關機。
4. **04 注入限制**：顧客沒有電腦、不熟設定、只有 7 天可驗收；因此需要低風險、可在家重複的基準流程。
5. **05 生成角度**：診斷篇、福利機驗收篇、更新後觀察篇、正常與異常判斷篇、何時聯絡賣家／送修篇。
6. **06 去重與查核**：若核心流程相同就合併；列 Apple 文件、機型／OS、實機測試條件與不能武斷承諾的門檻。
7. **07 形成提案**：〈福利 iPad 待機掉電正常嗎？收到後 7 天建立續航基準的檢查流程〉。
